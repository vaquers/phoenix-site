# Логика контроля доступа

## Роли

| Роль        | user_id     | Описание |
|------------|-------------|----------|
| Супер-админ | 5584466914  | Единственный владелец. Управляет списком пользователей и флагом is_admin. |
| Админ       | is_admin=True в БД | Обычный админ (права задаются супер-админом). |
| Пользователь | Запись в users | Разрешённый пользователь, может пользоваться ботом. |
| Не в списке | Нет в users | Доступа нет: бот не сохраняет таких пользователей и не даёт функционал. |

## Поток управления доступом

1. **Входящее обновление (сообщение / callback)**
   - Middleware проверяет: пользователь есть в таблице `users` ИЛИ это супер-админ ИЛИ это команда `/start`.
   - Если нет — запрос не передаётся в хендлеры, отправляется сообщение «Нет доступа» (кроме `/start`: на `/start` хендлер сам отвечает «Нет доступа» и не добавляет пользователя).
   - Супер-админ всегда пропускается middleware (может писать боту даже до «официального» добавления в БД, если нужно).

2. **Команда /start**
   - Если пользователя нет в `users` → ответ «У вас нет доступа к боту».
   - Если пользователь есть → приветствие и выдача клавиатуры (у админа — админская, у обычного — пользовательская).

3. **Управление доступом (только супер-админ)**
   - Добавление: супер-админ отправляет команду вида `/add_user 123 456 789` → в `users` добавляются эти user_id с `is_admin = false`.
   - Назначение админа: `/set_admin 123 1` (сделать админом), `/set_admin 123 0` (снять).
   - Пользователи в БД создаются только этими действиями; автоматического сохранения по факту «написал боту» нет.

4. **Проверки в коде**
   - Единая точка: «есть в users» проверяется в middleware и при необходимости в хендлерах через общий фильтр/функцию.
   - Супер-админ проверяется по константе `SUPER_ADMIN_ID` (в config).

## Схема (упрощённо)

```
[Сообщение] → Middleware:
                ├─ /start? → пропустить в хендлер
                ├─ супер-админ? → пропустить
                └─ user_id в users? → пропустить, иначе «Нет доступа»

/start handler:
                ├─ user_id не в users → «Нет доступа», конец
                └─ user_id в users → приветствие + клавиатура (админ/пользователь)

/add_user, /set_admin:
                ├─ отправитель != супер-админ → игнор или «Нет прав»
                └─ супер-админ → выполнить добавление/установку is_admin
```

## Реализация (aiogram)

- **config**: константа `SUPER_ADMIN_ID = 5584466914`.
- **db.queries.user_queries**: `get_user(user_id)`, `add_user(user_id, is_admin)`, `set_admin(user_id, is_admin)`, `get_allowed_user_ids()` (для middleware).
- **Middleware**: перед хендлерами проверять: событие — `/start` ИЛИ from_user.id == SUPER_ADMIN_ID ИЛИ from_user.id в списке из `get_allowed_user_ids()`. Иначе — ответ «Нет доступа» и не вызывать handler.
- **Фильтр**: например `AllowedUserFilter` / функция `is_allowed(user_id)` для переиспользования в хендлерах при необходимости.
- **Супер-админ**: фильтр `SuperAdminFilter` (message.from_user.id == SUPER_ADMIN_ID); обработчики `add_user` и `set_admin` регистрируются только с этим фильтром.
- **Старт**: в хендлере `/start` только проверка «есть в users» и вывод приветствия или «Нет доступа», без добавления в БД.
